Science, programming & zombie projects: Lessons I have learnt

2015-09-25
by Paul Agapow
in science

tagged bioinformatics, programming, software, langauges

I am my own worst enemy.

IF SIMPLE THEN MAKE_MORE_COMPLICATED

Some years ago, someone asked me for a simple program, with the caveat that it needed a point-and-click user interface. I agreed and thought it was a good opportunity to use RealBasic. Don't laugh: it let me draw the user interfaces out quickly, worked fast enough for the task and compiled versions for multiple platforms (Mac, Windows, Linux) that looked and behaved like proper native apps.

Said little app was sucessful and more features were called for. (You can see where this is going, right?) Then there was more "success" and more calls for features. Eventually it was manipulating and editing forests of phylogenetic trees according to the results of a simulated annealing optimization, with operations for walking the tree and saving the results in multiple formats ...

Needless to say, this was not really what RealBasic was for. In any event, RB ran on a yearly license / edition model, so I had to keep updating my subscription to keep up with bug fixes and OS updates. Furthermore, RealBasic got bought out or consumed by another company and the product changed name and evolved into something else. The project format was some proprietary hogwash so you couldn't use your own editor, let alone extract the source code in an attempt to get it into another system or translate it.

And so I'm locked into this shambling corpse of a development system ...

Multiply-defined symbol: "what was the point of this"

Another time, I wrote a program that pereformed complex evolutionary calculations. Given the time of authoring, this was done in C++ (for speed, and because scripting langauges weren't quite there and weren't quite fast enough as yet). Although the design was nicely modular and extensible with a wide set of features (even as the author, I can say it was a good piece of work), demands for even more features piled in, that required minor alterations or combinations of previous ones. Add this evolutionary model, add this manipulation, alter these parameters, impose this sort of extinction regime which depends on this sort of evolution on these sort of organismal traits ...

It would have been simple if the whole thing had been in a scripting language. But it wasn't and I had this huge mass of complex simulation code locked up in an irreducible mass of stuff. Debuggingg evolutionary simulations was its own sort of fun, especially as organisms tended to evolve to exploit bugs or edges cases in the code ...

And so I'm locked into this zombie design ...
